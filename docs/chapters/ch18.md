## Motivation

No near-term — and arguably no long-term — useful computation is purely quantum. Quantum processors will be accelerators within classical systems, much as GPUs are today: invoked for the sub-tasks they accelerate, embedded in classical data pipelines, orchestration, and user interfaces. This chapter studies the *architecture* of such hybrid systems, the very pattern this repository's API embodies.

A hybrid system is not simply a Python script that calls a quantum simulator. It is a set of contracts: how data are represented, where validation happens, how jobs are scheduled, which results are reproducible, how uncertainty is reported, and how the service behaves when a backend is slow or unavailable. The scientific algorithm and the software architecture cannot be separated. A beautiful quantum subroutine that requires millions of serial cloud calls may be unusable; a careful batching strategy can make a modest subroutine practical.

Hybrid computing also clarifies the most important conceptual divide in QML. Classical data must be loaded into a quantum device, often through costly state preparation. Quantum-native data may already exist as states or processes produced by sensors, experiments, or simulators. The latter changes the economics: the goal may be to extract useful information without first converting an exponentially large quantum object into a classical description.

## Theory

### Partitioning work by comparative advantage

A hybrid quantum-classical system partitions work by comparative advantage. The quantum processor prepares states, evolves them, and measures; the classical processor handles optimisation, data management, error mitigation, and control (Chapters 7, 17). The interface between them — latency, bandwidth, and the cost of moving between quantum and classical representations — is often the true performance bottleneck, not the quantum hardware itself. Tight feedback loops, as in real-time decoding (Chapter 8), demand co-located classical compute; loose loops, as in molecular screening (Chapter 10), tolerate cloud round-trips.

The quantum side typically provides one of three outputs:

- **Expectation values**, such as $\langle\psi(\theta)|H|\psi(\theta)\rangle$ for VQE.
- **Samples**, such as bitstrings from QAOA or a generative circuit.
- **Fidelities or overlaps**, such as quantum-kernel entries.

The classical side turns those outputs into decisions: optimiser updates, model scores, confidence intervals, API responses, database records, or alerts. Most production complexity lives in this classical layer.

### Hybrid loops

The archetypal loop evaluates a quantum function inside a classical algorithm:

```text
client request
     │
     v
validate schema ──> choose simulator/hardware backend ──> submit circuits
     │                                                   │
     │                                                   v
     │                                         shots, counts, metadata
     │                                                   │
     v                                                   v
classical post-processing <── error mitigation / aggregation
     │
     v
optimiser step, prediction, or typed API response
```

In VQAs, the classical optimiser proposes parameters $\theta$ and the quantum device evaluates a cost $C(\theta)$. In kernel QML, the classical service builds pairs of data points and the quantum device estimates overlaps. In sensing, a quantum device may produce states whose measurement settings are selected adaptively by a classical controller. These are all hybrid systems, but their latency tolerance differs dramatically.

### Latency and throughput

**Latency** is the time from requesting a quantum evaluation to receiving a usable result. **Throughput** is the number of evaluations completed per unit time. Quantum cloud services often have high latency because jobs must be queued, compiled, calibrated, executed, and returned. Even simulators can be expensive when circuits or batch sizes grow.

A practical architecture therefore avoids unnecessary serial calls:

- Batch independent circuits and parameter points.
- Cache deterministic simulator results and repeated kernel entries.
- Separate interactive endpoints from long-running job workflows.
- Record backend, transpilation, shot count, seed, and calibration metadata.
- Prefer asynchronous execution for workloads that exceed request-response time budgets.

A tight feedback loop with parameter-shift gradients may require two shifted evaluations per parameter, per optimisation step. If each evaluation is a separate remote job, latency can dominate. Batching the shifted circuits, using simultaneous perturbation methods, or moving optimisation closer to the backend can change the feasibility of the entire algorithm.

### Classical data versus quantum-native data

A deeper distinction governs where quantum advantage is most plausible: the nature of the *data*. **Classical data** loaded into a quantum computer must pass through a state-preparation bottleneck, and the dequantisation results of Chapter 16 caution against expecting generic speedups. Loading a vector into amplitudes, for example, can hide substantial cost in the assumption that state preparation is efficient.

**Quantum data** are states, channels, or measurement records produced by quantum systems: a sensor array probing a field, a material simulator producing many-body states, a communication channel with unknown noise, or a laboratory experiment generating entangled states. Learning directly from such data may avoid full classical tomography. Theory shows that for some tasks, quantum models can characterise quantum processes from fewer experiments than any classical strategy. This does not make every quantum-data task advantageous, but it identifies a more natural domain for quantum learning.

A system architect should therefore ask:

1. Where does the data originate: a database, a sensor, a quantum device, or a simulator?
2. Is the quantum state preparation part of the workload, or is the state already present?
3. What classical output is actually required: a label, an expectation, a certificate, or a full description?
4. Which classical baseline receives the same input access and accuracy target?

### Software stacks

Hybrid systems sit on layered software stacks. At the top are APIs, notebooks, workflow engines, and model-training code. Beneath them are quantum SDKs such as PennyLane, Qiskit, Cirq, Braket, or vendor runtimes. Beneath those are transpilers, pulse schedulers, simulators, device queues, calibration systems, and hardware control software.

The same algorithm may be expressed differently at each layer. A researcher may write a high-level VQE circuit; a compiler decomposes it into native gates; a runtime batches parameter values; a device returns counts; a service converts counts into an energy with uncertainty. Production design requires clear boundaries so that an improvement in one layer — for example, a better transpiler or simulator — does not silently change scientific meaning.

### Reliability and observability

Engineering maturity matters as much as theory. A production hybrid service needs validated inputs, graceful degradation when the quantum backend is unavailable, reproducible environments, observability, and clear API contracts. The companion service in this repository demonstrates these patterns: thin, validated web routes wrap thoroughly tested library functions, so the science is verifiable independently of the deployment, and classical simulation provides a faithful fallback during development.

A robust service should expose not only a number, but also context:

- Was the result simulated or obtained on hardware?
- Which backend, seed, shot count, and compilation options were used?
- What is the estimated statistical uncertainty?
- Which version of the model or circuit generated the result?
- What fallback path was used if the preferred backend failed?

Without this metadata, hybrid computation becomes difficult to debug and impossible to audit.

## From theory to code

The FastAPI service is the hybrid architecture in miniature: each endpoint
validates a request, dispatches to a tested `aiquantum` function, and returns a
typed response.

```python
from fastapi.testclient import TestClient
from aiquantum.api.app import app

client = TestClient(app)
r = client.post("/api/v1/qml/kernel", json={"x_a": [[0.0], [0.0]]})
print(r.json()["kernel_matrix"])   # 2x2 quantum kernel
```

The API reference for these routes is maintained in `../api.md`. The request above is local and testable: no external quantum service is contacted. That design makes the scientific core deterministic during development while leaving room for future hardware-backed implementations behind the same contract.

### Request flow through a hybrid API

```text
POST /api/v1/qml/kernel
        │
        v
Pydantic/FastAPI validation
        │
        v
route handler selects aiquantum.qml.quantum_kernel_matrix
        │
        v
state-vector simulation computes kernel entries
        │
        v
JSON response with matrix, status code, and client-visible contract
```

The same architecture can wrap real hardware, but the service boundary should then include job IDs, asynchronous polling, shot counts, backend names, and uncertainty estimates. A synchronous endpoint is appropriate for small deterministic simulations; long hardware runs usually require a job resource rather than a blocking HTTP request.

### PennyLane and PyTorch hybrid layer

```python
import pennylane as qml
import torch

n_qubits = 2
dev = qml.device("default.qubit", wires=n_qubits)


def circuit(inputs, weights):
    qml.AngleEmbedding(inputs, wires=range(n_qubits), rotation="Y")
    qml.StronglyEntanglingLayers(weights, wires=range(n_qubits))
    return [qml.expval(qml.PauliZ(i)) for i in range(n_qubits)]


qnode = qml.QNode(circuit, dev, interface="torch", diff_method="parameter-shift")
weight_shapes = {"weights": (2, n_qubits, 3)}
quantum_layer = qml.qnn.TorchLayer(qnode, weight_shapes)

model = torch.nn.Sequential(
    quantum_layer,
    torch.nn.Linear(n_qubits, 1),
    torch.nn.Sigmoid(),
)

x = torch.tensor([[0.1, 0.4]], dtype=torch.float32)
print(model(x))
```

Here PyTorch owns batching, loss functions, optimisers, and model composition; PennyLane owns quantum evaluation and gradients. This is a genuine hybrid stack even though the device is a simulator. Replacing `default.qubit` with hardware would introduce finite shots, latency, and operational constraints.

### Classical optimisation meets quantum evaluation

A useful mental model is a database transaction with an expensive external call. The optimiser cannot inspect the quantum state directly; it receives scalar estimates, samples, or gradients. The quantum backend does not know the optimiser's history; it executes circuits. Good hybrid software narrows this interface and makes it explicit.

For example, a VQE service might accept a Hamiltonian identifier and ansatz configuration, launch an optimisation job, store intermediate energies, and return a final estimate with metadata. A kernel service may accept feature vectors, compute a symmetric Gram matrix, and return it to a classical SVM. A sensing service may accept a measurement policy and stream adaptive settings back to lab hardware. In each case, classical and quantum responsibilities should be independently testable.

## Summary

Quantum processors are accelerators within classical systems. The interface and the *origin of the data* — classical versus quantum — determine where advantage lives, and sound software engineering determines whether it can be delivered. Hybrid loops combine classical optimisation, validation, orchestration, caching, and observability with quantum evaluations that may be simulated or hardware-executed. Robust API contracts, such as those described in `../api.md`, make the boundary explicit and keep scientific computation reproducible.
